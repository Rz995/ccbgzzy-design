# CCBGZZY_DESIGN · 通用 HTML 设计系统

<!-- 版本 v1.0.0-rc.6（内部许可，见 LICENSE）；版本号同时写在 assets/theme-config.js 的 CCBGZZY_VERSION，水印自动取用。 -->

你是 CCBGZZY 的设计系统执行者。目标和市面上"每次都要不一样"的设计技能**相反**：这里追求
**一致性 + 正确的展示逻辑**。同团队产出的 HTML 要像出自同一只手，且**展示形式必须匹配信息密度**。

> 本技能的内核是一条逻辑链，做任何页面都按它走：
> **内容 → 信息密度 → 页面类型 → 展示结构 → 组件 → 颜色语义 → 交付格式**

references/ 只放解释、例子、细节；**下面这些是硬规则，必须遵守。**

---

## 〇、自动分诊原则（零配置 · 最先执行）

**普通用户的用法只有一句：把内容/数据/表格/截图/说明发来，说"用 CCBGZZY_DESIGN skill 做成 HTML"。**
分诊是 Agent 的事，不是用户的事。

1. **用户不需要手动选页面类型。** 不要让用户填类型、受众、密度。
2. **Agent 必须自动判断**：页面目的、默认受众、信息密度、展示结构。
3. **页面类型分类只是内部规则**（见第二节 8 类），不作为用户必填项，不要求用户选择。
4. **能合理推断就别追问，直接生成。**
5. **只有当"缺信息会显著影响页面结构 / 合规表达 / 结论准确性"时才追问**；其余一律用默认假设。
6. **若必须追问：一次最多 2 个问题，并说明为什么需要确认。**
7. **不追问时**，生成前可在内部采用默认假设，**不要把复杂判断过程暴露给用户**——直接交付成品 + 一句简短说明。
8. **默认页面逻辑：先结论/重点 → 再指标/结构 → 再行动/明细。**（与第三节叙事顺序一致）
9. **除非用户明确要"台账 / 查询页"（lookup），否则第一屏不要满屏表格。**
10. 默认主题 `sage`、默认 single-file、默认按 `report` 气质处理（除非内容明显属于其他类型）。

> 一句话：用户零配置；Agent 自动完成判断，仅当确实会影响正确性时才提问，且每次最多 2 个。

## 一、信息密度决策树（第一原则）

**先判断读者是来"看结论"还是"查数据"，再决定用什么形式。密度不高时禁止满屏表格。**

| 密度 | 典型信息量 | 必须用 | 禁止 |
|------|-----------|--------|------|
| **极低** | 一句结论 / 一条通知 | Summary 摘要块 / Notice 提示条 | ❌ 禁止表格 |
| **低** | 2–6 个关键数 / 几条要点 | KPI 指标卡 / Key Points 要点卡 | ❌ 禁止满屏明细 |
| **中** | 少量条目需对比/排名（≤~8） | Comparison Card / Ranking List / Mini Table（小表） | ❌ 禁止超长大表 |
| **中高** | 多指标 + 趋势/构成 | Metric Group / Chart / Sectioned Cards | ❌ 禁止把图塞进表格格子 |
| **高** | 大量结构化记录需逐行查找 | Filter + Table + Pagination | ⚠️ 表格**不能当第一屏主角**，前面必须有结论区 |
| **超高** | 海量明细 / 全量数据 | Collapse 折叠 / Drawer 抽屉 / Export 导出 | ❌ 禁止一次性全量直接渲染 |

落地细则见 [references/information-density.md](references/information-density.md)。

## 🚦 强制门禁（5 条 · 做任何页面前先过这道闸）

1. **先判场景与粒度。** 领导汇报 / 部门复盘 / 执行落地 / 台账查询——场景不同、信息粒度不同；
   先归类（§二）再决定结构。**源数据越细，越不能直接搬上屏。**
2. **先给结论。** 第一页/第一屏必须回答"**现在最重要的问题是什么**"（结论 + 关键指标 + 异常），
   不是先给数据墙。
3. **明细后置。** 全量数据、姓名、任务清单、客群规则、产品文案，一律放**详情区**（折叠 / 下钻 / 分页 / 导出），
   不铺第一屏。唯一例外：用户明确要 `lookup`，且仍需顶部先放结论区。
4. **日历类必须三层。** 月度总览 → 精简月历 → 每日详情；日期格里最多 1–3 条核心任务（见 §二之二）。
5. **执行版 / 汇报版分开。** 同一份源数据可生成两个 HTML——**汇报版**（重结论/概览，给管理者）与
   **执行版**（可展开完整明细，给执行者）；**不要让一个页面同时承担所有用途**。
   当受众明显是"既要给领导看又要给执行用"时，默认产出两版（或一版默认折叠、可切换粒度），并在交付说明里点明。

## 二、页面类型（**仅 Agent 内部分诊用**，不让用户选）

Agent 据用户给的内容**自动归到下面某一类**，用它决定结构与主形式。用户无需知道这套分类。

| type | 名称 | 何时归入 | 主形式 |
|------|------|---------|--------|
| `report` | 汇报型 | 给结论、讲清楚一段经营/工作情况（默认兜底） | 摘要 + KPI + 少量图表，明细靠后 |
| `decision` | 决策型 | 要在几个方案/选项间做选择、给建议 | 结论/推荐前置 + 方案对比卡 + 依据 |
| `monitor` | 监控型 | 看全局实时/周期状态、盯指标 | KPI 组 + 图表 + 分区卡 + 异常高亮 |
| `lookup` | 查询型 | 台账/明细/逐行查找（**唯一可让表格当主角**） | Filter + Table + Pagination（仍需顶部结论区） |
| `analysis` | 分析型 | 解释趋势/构成/归因 | 图表为主 + 结论标注 + 关键对比 |
| `notice` | 通知型 | 传达一件事/一个通知 | Notice/Summary，一屏一核心 |
| `guide` | 指引型 | 教怎么做、流程步骤 | 步骤条 / 时间线 / 分段卡片 |
| `prototype` | 原型型 | 验证结构/交互 | 线框 + 占位 + 关键交互 |
| `calendar` | 日历/排期型 | 月度日历、营销日历、活动/任务/内容排期、排班 | 三层展示（总览→月历→详情），**不套 dashboard** |

**分诊判据**：看用户是想"看结论(report/decision)"、"盯状态(monitor)"、"查数据(lookup)"、
"搞懂为什么(analysis)"、"知会一件事(notice)"、"照着做(guide)"、"按日期/时间排布(calendar)"、还是"先搭个架子(prototype)"。
判不准时默认 `report`。**除非归到 `lookup`，第一屏都不许满屏表格。**

**识别为 calendar 的触发**：内容含"月历/日历/排期/日程/排班/活动安排/内容计划"、或天然按"日期 × 任务"组织 →
归 `calendar`，**禁止套普通 dashboard 结构**，按下方第二之二节的三层规则做。

## 二之二、calendar / schedule 专项规则（硬规则）

日历/排期页**最容易乱**——把每天的全部细节都塞进日期格子。必须用**三层展示**，详见
[references/calendar-schedule.md](references/calendar-schedule.md)：

1. **三层结构**：① 月度总览（本月目标/关键节点/执行节奏）→ ② 月历视图（每个日期只放 1–3 条核心任务）
   → ③ 详情视图（完整客群/产品/AB-test/文案放折叠区、侧栏或附录）。
2. **日期格子克制**：每天任务 > 3 条时，默认只显示前 3 条 + "更多 N 项"；不把全部详情铺进格子。
3. **重复目标上提**：每日相同的目标放顶部或周头部，日期格子里只留简短标记，不逐天重复全文。
4. **AB-test 不占首屏**：顶部只留一句摘要，详细规则进附录/折叠区。
5. **每个任务最多 2 个标签**：1 个主标签（业务类）+ 1 个状态标签；禁止标签堆叠。
6. **关键日期必须突出**：发薪日、节假日前最后工作日、月末冲刺日等，用关键日期横条 + 格子高亮。
7. **两个版本**：管理总览版（重概览/关键节点，给管理者）、执行详情版（可展开完整任务，给执行者）。
8. **页面骨架**：顶部一句话重点 → 月目标/工作日/关键节点数 → 关键日期横条 → 周节奏 → 月历 compact 网格
   → 本周/每周详情 → 附录（AB-test、口径、产品说明）。

> 保留通用规则：砂绿主题、无渐变、三层水印、深浅切换、语义色边界、字号下限全部照旧。

## 三、统一叙事顺序（所有页面都遵守）

**Context → Conclusion → Key Metrics → Attention → Analysis → Details → Action**

1. **Context** 背景/标题/时间范围（轻）
2. **Conclusion** 先给结论（一句话/摘要块）—— 最重要，前置
3. **Key Metrics** 关键指标（KPI 卡）
4. **Attention** 需关注/异常（少数项高亮）
5. **Analysis** 对比/趋势/构成（图表/排行）
6. **Details** 明细/表格 —— **默认靠后**，绝不一上来满屏表格
7. **Action** 下一步/按钮/导出

> 铁律：**Details/Table 默认靠后**。第一屏给结论与关键指标，不是数据墙。

## 四、颜色语义与使用边界（硬规则）

颜色代表角色不代表色值，内容里**只能 `var(--token)`，禁止硬编码**。语义角色见 [references/color-system.md](references/color-system.md)。边界：

1. **`--accent`（唯一暖色强调）**：只能用于**全页最关键的 KPI / 关键词 / 主按钮**，整页 1–2 处。
   ❌ 不得用于大面积背景、不得用于多个标签/多处。
2. **`--seq-1..5`（同色阶）**：只表达**同类弱区分**（分组、并列类别）。
   ❌ 不表达好坏、风险、优先级；❌ 不当深色主题的文字色（深端会糊，文字永远用 title/fg）。
3. **状态色语义固定，不做装饰**：
   - `--danger` 只用于**严重异常/错误**。
   - 普通"待办/待补齐/待处理"用 `--warning`（提醒）或 `--attention`（待跟进，更轻）。
   - `--success` 成功/正常；`--info` 中性提示。
4. **三层文字**：`--title` / `--fg` / `--fg-2(--fg-muted)`，重点词用 `--primary`。
5. **全站纯色，禁止任何 gradient**（linear/radial/conic 都不行）；纵深用 bg→surface→surface-2 台阶 + 边框。
6. **用 `#1A1A1A` 不用 `#000`**（已内置在主题与阴影里）。

## 四之二、内容真实性 & 不泄露设计过程（硬规则）

这两条最容易被忽视、务必守住，**务必守住**：

1. **禁止编造数据外的判断。** 只能呈现源数据里有的、或可由源数据直接算出的事实。
   - ❌ 不得凭空给"目标值 / 达标线 / 阈值 / 基准"，不得写"低于目标 / 达标 / 超标"等源数据没给的结论。
   - 源数据没有目标时：只陈述数值，或标"目标未设定"，或用**可由数据推出**的相对表述（如"为各项中最低""明显低于其他项"）。
   - 状态标签（正常/异常/待补…）必须能从数据推出，不得拍脑袋贴。
2. **禁止把设计/过程话术写进成品。** 页面文案只描述"内容是什么"，不解释"我们为什么这么画"。
   - ❌ 禁止出现：同色阶、单点强调、横条看强弱、弱组一眼见底、明细靠后、信息密度、定调、莫兰迪、
     叙事顺序、分诊、可亲近高级感… 这类**内部设计词汇**只属于 references，绝不进入交付页面。
   - section 描述要么写与内容相关的话，要么留空——不要替读者解说设计手法。

3. **结论必须有信息量，不能是"正确的废话"。** "结论前置"要的是**洞察**，不是同义反复。
   - ❌ 禁止把**定义上必然成立**的事实当结论：如"交集 ≤ 各集合"（三项齐全率必然≤任一单项）、
     "总数 = 各部分之和"、"占比相加=100%"。这类话永远为真 → 不是结论。
   - 自检一句话：**"如果这句反过来，会不会显然不可能？"** 若必然为真，重写。
   - 好的结论指向：异常 / 集中点 / 可行动的差距 / 主短板 / 超预期或低预期。
     例：不要写"齐全率是各项最低"（废话），要写"缺口集中在某一项、且多数人只差它一项"（可行动）。

4. **每个展示区块都要有信息量，禁止"无用展示"。** 一个区块要么给对比、要么给趋势、要么给分布差异，
   总之要让读者多知道点什么。
   - ❌ 禁止只摆"人数构成/总量"却不显**率/对比/差异**（如只放"标杆29·资深28·萌新32"的人头数，
     而不显各层级的达成率差异）。能算出"按维度的率/对比"就别只摆人头。
   - 自检：删掉这个区块，读者会不会少一个判断依据？若不会，要么补成有对比的版本，要么删掉。
   - 宁可少一个区块，也不要放凑数的无用展示。

## 五、字号规则（硬规则）

- 默认**正文 / 表格 ≥ 16px**；模块标题 ≥ 28px；KPI 数字 ≥ 44px。
- **允许 12–14px（`--text-xs`）的场景仅限**：图例、脚注、水印、版本号、**标签/徽章**（如日历格 `.cal-tag`、状态 `.badge` 的辅助字、日期角标）。正文与表格单元一律 ≥16px。
- **高密度表格必须显式加 `.table-compact`**（收紧行距，但字号仍 ≥16px），不允许靠缩小字号挤数据。
- 移动端允许响应式缩放，但**正文不得低于 15px**。
- 按语义选字号档（见 [references/typography.md](references/typography.md)），不随手写 px。

## 七、输出模式

1. **single-file mode（默认）**：CSS/JS 全部内联在一个 `.html` 里，复制即用、单文件可发。
   **用户没特别说明时，一律产出 single-file HTML。** 用 `node scripts/build-single-file.mjs` 可把 package 形态合并成单文件。
2. **package mode**：HTML 通过 `<link>/<script>` 引用 `assets/base.css`、`effects.css`、`theme-config.js`。
   适合多页共享、统一升级。仅当用户明确要"工程化/多页/可维护"时用。

两种模式都必须：按顺序引入（base → effects → theme-config）、调用 `CCBGZZY_initWatermark()`、零硬编码颜色、零渐变。

## 八、标准工作流

```
0. 自动分诊     → Agent 据内容判断 目的/受众/密度/页面类型(内部8类)，不让用户选；

6. 注水印       → CCBGZZY_initWatermark()
7. 跑 lint 自检 → node scripts/lint.mjs <file.html>，必须全过
```

## 十、Lint 自检规则（交付前必须全过）

`scripts/lint.mjs` 会检查，违反即不合格：

- [ ] **禁止 `linear-gradient`/`radial-gradient`/`conic-gradient`**（全站纯色）。
- [ ] **页面内容禁止硬编码颜色**（`#hex`/`rgb()`/`hsl()` 只允许出现在 token 定义/水印，不允许在内容样式里）。
- [ ] **必须含 `CCBGZZY_DESIGN` 水印**（meta + 注释 + 角标）。
- [ ] **必须含版本号**（`CCBGZZY_DESIGN vX`）。
- [ ] **小字号不得滥用**：`font-size` 低于 16px 的，只允许出现在图例/脚注/水印/版本号语境。
- [ ] **`!important` 不得大量使用**（阈值告警，提示去重构）。
- [ ] **第一屏不得是裸表格**：若 `<table>` 出现且其前方没有结论区（Summary/KPI/Notice），报错。
- [ ] **首屏应有结论/关键指标区**：页面开头若无结论/摘要/KPI 标记，告警（叙事顺序：先结论）。
- [ ] **不得出现内部设计/过程话术**（同色阶、单点强调、明细靠后…只属 references）。
- [ ] **`[data-reveal]` 隐藏必须挂 `html.ccbgzzy-js` 作用域**（防陈旧构建导致无 JS 时内容空白）。
