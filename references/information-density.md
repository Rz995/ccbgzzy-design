# 信息密度 → 展示形式

> 这是"展示逻辑"的总纲。一句话：**展示形式由信息的"形态 + 密度 + 阅读目的"决定，不是默认堆表格。**
> 密度不高时一上来就糊一张满屏表格，会让读者很累——因为表格是"逐行查找/对比"的工具，
> 当信息其实只是几条结论时，表格是错的形式。

## 一、先判断：读者来这屏是要"看结论"还是"查数据"？

- **看结论**（密度低）→ 用大数字、卡片、留白叙事，一屏一个核心。
- **查数据**（密度高、行多、需逐行比对）→ 才用表格，且按 tables.md 做清爽。

大多数管理/汇报页面是"看结论"，却被默认做成"查数据"的满屏表格——这是最常见的累。

## 二、形态 → 形式 对照表（核心）

| 信息形态 | 推荐形式 | 不要 |
|---------|---------|------|
| 1 个关键数字 / 结论 | 大数字 stat（Hero/KPI 大字）+ 一句话 | 塞进表格某个格 |
| 2–6 个并列指标 | 指标卡组（`.kpi` 网格） | 一行小字罗列 |
| 少量条目（≤ ~8 条），每条几个字段 | 卡片网格 / 清单列表 | 上来就表格 |
| 大量结构化记录（行多、字段规整、要逐行查找比对） | 表格（表格此时才该出场） | 拆成几十张卡片 |
| 排名 / 强弱对比 | 横向条形 / 排行榜 | 一堆数字网格 |
| 占比 / 构成（≤5 类） | 单条堆叠条 / 环图 | 多色饼图堆一屏 |
| 趋势 / 时间序列 | 折线 | 表格里塞一排数字 |
| 流程 / 步骤 | 步骤条 / 时间线 | 编号段落 |
| 长文 / 叙述 | 卡片化分段、留白叙事 | 一大块文字平铺 |

## 三、信息密度三档（density tiers）

| 档位 | 何时用 | 做法 |
|------|--------|------|
| **宽松 spacious** | 密度低、给人看结论（汇报首屏、概览） | 大留白、大字、卡片、一屏一核心；留白可达 30%+ |
| **舒适 comfortable**（默认） | 概览 + 适度明细 | 卡片 + 关键指标 + 必要的小表格 |
| **紧凑 compact** | 密度高、给人查数据（明细页、后台） | 表格、紧凑行距，但仍守最小字号（正文/表格 ≥16px） |

> 同一份数据，概览屏用宽松、明细屏用紧凑——分层，而不是一屏全塞。

## 四、五条展示逻辑铁律

1. **先结论后明细（summary-first / 渐进披露）。** 先给概览、重点、总指标；明细折叠、分页、
   或放到下一屏/可展开区。别让读者一上来面对一堵数据墙。
2. **密度不高别硬塞表格。** ≤ ~8 条、字段少 → 卡片或列表。表格留给"行多、要逐行查找对比"。
3. **一屏一核心。** 每屏只突出一个主信息，其余收为辅助层（次级颜色、更小权重）。
4. **给喘息。** 留白 ≥ 20%；用分组、分段、卡片把信息切块，别平铺一大片。
5. **强弱分明。** 同屏里"最该看的"用大字/`--accent` 点 1–2 处，其余用 `--fg-2`/`--fg-muted` 退后。

## 五、容器容量预算（硬规则）

小容器不能承载完整说明句。小容器包括：日期格、KPI 小卡、标签、徽章、按钮、导航项、表头、角标。

| 容器 | 允许承载 | 不允许 |
|------|----------|--------|
| 日期格 | 日期、短标记、0-1 条核心事项、`+N 项` | 完整执行清单、长文案、多字段说明 |
| KPI 小卡 | 指标名、数字、短辅助说明 | 长解释、多段文字、复杂口径 |
| 标签 / 徽章 / 角标 | 类别、状态、节点类型中的一种 | 同时表达类别+状态+说明 |
| 按钮 / 导航项 / 表头 | 短命令、短标题、字段名 | 多行动作、完整句子 |

超过容量时必须做结构变化：

1. 上提到父级摘要或周节奏。
2. 压缩为 `+N 项`。
3. 进入折叠、抽屉、详情层或附录。
4. 改用列表、时间轴、周视图或可筛选表格。

禁止用缩小字号、压缩行距、强制多行换行来解决信息过载。小容器装不下，说明结构错了。

## 六、换行契约

### 禁止换行区域

以下区域默认 `white-space: nowrap; overflow: hidden; text-overflow: ellipsis;`，最多 1 行：

- 日期格内任务标题。
- 标签 / 徽章 / 状态角标。
- 按钮 / 导航 / 表头。
- KPI 数字和单位。
- 卡片标题第一行。

### 允许换行区域

以下区域可以换行，但必须控制行数：

- 摘要段落。
- 详情说明。
- 每日执行详情。
- 折叠面板正文。
- 表格移动端卡片正文。

普通说明最多 2-3 行；长说明进入详情层。

### 换结构触发

出现下列任一情况，说明当前容器不适合继续承载：

- 一个日期格需要超过 1 行任务文案。
- 一个日期格里出现 2 个以上标签。
- 连续多天都需要换行。
- 同一日任务超过 1 条且都需要解释。

触发后，月历不能作为主展示，必须改为"周节奏 / 时间轴 / 列表 + 轻量月历"，或在执行场景改为周视图、日期列表、可筛选任务表。

## 七、把"满屏表格"改轻的常用手法

- **概览 + 明细分层**：首屏给几张指标卡 + 一句结论；表格收到下方或"查看明细"里。
- **表格瘦身**：只留主标识列 + 1–2 个关键列，其余次要列折叠/隐藏（窄屏尤其）。
- **表格换形式**：若每行其实只在比一个指标的高低 → 换成横向条形/排行榜，比满屏数字清爽。
- **分组归拢**：长表按类别分组，用分组小标题行断开，给眼睛落点。
- **重点前置**：把"异常/最高/最低/需关注"的少数项单独提到顶部做高亮卡，其余收进表格。

## 八、移动端阅读优先（≤760px 硬约束）

手机屏窄，密度判断要更狠一档——**正文第一，工具层让位**：

1. **工具层不抢正文。** 标题栏、主题切换、侧栏、按钮都收进 appbar/紧凑控件/抽屉（见 components.md），
   不许横铺、不许把正文往下挤、不许撑出横向滚动。主题切换收成 pill/图标，不是一排色卡。
2. **标题永远一行。** 移动端标题用 `.ccbgzzy-title-oneline`（省略号），长标题给 `data-mobile-title` 短版；
   长结论交给正文摘要块，标题栏只放短标题。
3. **表格一律转卡片，不横向拖拽。** 字段 >~6 个时把次要列 `data-mobile-hide` 收进展开详情（见 tables.md）。
4. **每屏更少核心。** 桌面"一屏一核心"，移动端进一步：一屏一块、其余靠滚动渐进披露。
5. **阅读时收起 chrome。** 滚动正文后 appbar 自动 compact、工具收 icon-only（headroom），把高度还给正文。

## 九、和组件的对应
指标卡 `.kpi`、卡片 `.card`、清单/列表、表格 `.table`（见 tables.md）、
排行条/进度 `.progress`、同色阶图例 `.seq-legend`（见 data-viz.md）、
工具层 `.ccbgzzy-appbar` / 紧凑主题控件 / drawer（见 components.md）。
先用本篇选对"形式"，再用对应组件去做。
